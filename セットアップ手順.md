# トト相撲 ONLINE セットアップ手順

`index.html` を GitHub Pages で公開し、Supabase で共有ランキングを有効にする手順です。所要時間は約15分、費用は無料枠内で収まります。

## 全体像

- 取組と結果は日付から決まるシード乱数で生成されるため、全プレイヤーが同じ取組・同じ結果を見ます(サーバー処理は不要)
- Supabase は「プレイヤーの成績(所持米・的中率)の共有」だけに使います
- 毎日15:30に投票締切、16:00に結果発表。リアルの本場所と同じ暦(奇数月・第2日曜初日・15日間)で進行します

---

## STEP 1: Supabase プロジェクト作成

1. https://supabase.com にアクセスし「Start your project」から無料アカウント登録(GitHubアカウントでログイン可)
2. 「New project」をクリック
   - Project name: `toto-sumo` など任意
   - Database Password: 自動生成のままでOK(控えておく)
   - Region: `Northeast Asia (Tokyo)` を選択
3. プロジェクト作成完了まで1〜2分待つ

## STEP 2: テーブル作成

1. 左メニューの「SQL Editor」を開く
2. 以下のSQLを貼り付けて「Run」を実行

```sql
-- プレイヤーテーブル
create table if not exists public.players (
  id uuid primary key,
  shikona text not null,
  ichimon text,
  title text,
  basho_key text,
  points integer not null default 15000,
  hits integer not null default 0,
  bets integer not null default 0,
  banzuke_level integer default 3,
  basho_count integer default 1,
  updated_at timestamptz not null default now()
);

-- Row Level Security(他人のデータを書き換えられないようにする)
alter table public.players enable row level security;

create policy "誰でも閲覧可" on public.players
  for select using (true);

create policy "自分の行のみ作成可" on public.players
  for insert with check (auth.uid() = id);

create policy "自分の行のみ更新可" on public.players
  for update using (auth.uid() = id);

-- ランキング取得用インデックス
create index if not exists players_basho_points
  on public.players (basho_key, points desc);
```

「Success. No rows returned」と出れば成功です。

## STEP 3: 匿名ログインを有効化

プレイヤー登録なしで遊べるようにします。

1. 左メニュー「Authentication」→「Sign In / Providers」(または「Providers」)
2. 「Anonymous Sign-Ins」を **Enabled** にして保存

※ これを忘れるとランキングに接続できません(ゲーム自体はオフラインで動きます)。

## STEP 4: 接続情報をゲームに設定

1. 左メニュー「Project Settings」→「API」(または「Data API」)を開く
2. 次の2つをコピー
   - **Project URL** (例: `https://abcdefgh.supabase.co`)
   - **anon public** キー (`eyJ...` で始まる長い文字列)
3. `index.html` をテキストエディタで開き、上のほうにある `CONFIG` を書き換える

```js
const CONFIG = {
  url: "https://abcdefgh.supabase.co",   // ← Project URL
  anonKey: "eyJhbGciOiJIUzI1NiIs..."     // ← anon public キー
};
```

> anon キーはブラウザに公開しても安全な設計のキーです(書き込み制限はSTEP 2のRLSが担保)。**service_role キーは絶対に貼らないこと。**

## STEP 5: GitHub Pages で公開

1. https://github.com で「New repository」→ 名前は `toto-sumo` など、**Public** で作成
2. 「uploading an existing file」から `index.html` をアップロードして Commit
3. リポジトリの「Settings」→「Pages」
   - Source: **Deploy from a branch**
   - Branch: **main** / **(root)** → Save
4. 1〜2分後、`https://<ユーザー名>.github.io/toto-sumo/` でゲームが公開されます

このURLを友達に共有すれば、全員が同じ取組を予想し、同じランキングで番付を争えます。

## STEP 6: 実データ運用（手動登録）

シミュレーションではなく実際の大相撲の取組・結果で遊ぶための設定です。**取組・番付・優勝三賞は「取組結果登録ツール」で手動登録**します（各サイトの利用規約・robots.txtに配慮した、白い運用にするため自動スクレイピングは行いません）。ニュース見出しだけはGoogleニュースの公開RSSから自動取得します。

### 6-1. データの手動登録（毎日の運用）

1. `取組結果登録ツール.html` をブラウザで開く（ローカルでOK・公開不要）
2. **手順1**：場所キー（例 `2026-09`）を入れ、いまリポジトリにある `data/basho.json` を読み込む（初日は無ければ空で新規作成）
3. **手順2（番付）**：場所の初日だけ、ABEMA TIMESの番付ページを貼り付けて登録（2日目以降はスキップ）
4. **手順3（取組結果）**：ABEMA TIMESの取組結果ページを貼り付け、その日を選んで登録
5. **手順4（優勝三賞）**：千秋楽後にABEMA等を見て手入力
6. **手順5**：`basho.json` をダウンロードし、リポジトリの `data/basho.json` に上書きしてコミット

> ゲームは `data/basho.json` を読み込み、今場所のデータ（番付20人以上）があれば **実データモード**(画面に🛰表示)、なければシミュレーション(🎲表示)で動きます。

### 6-2. ニュースだけ自動（任意）

「大相撲ニュース」を場所予想タブに表示したい場合のみ、以下の2ファイルを追加します。

- `scraper/fetch_basho.mjs` ← 同梱の `fetch_basho.mjs`（ニュース専用版）の中身をコピー
- `.github/workflows/update-data.yml` ← 同梱の `update-data.yml`（ニュース専用版）の中身をコピー

1日1回（JST 07:15）GitHub ActionsがGoogleニュースの「大相撲」RSSを取得し、`data/basho.json` の **`news` フィールドだけ**を更新します。手動登録した取組・番付・優勝三賞には一切手を触れません（手動更新と時間帯が重ならないよう朝に実行）。不要ならこの2ファイルは追加しなくてOK（ニュースカードが表示されないだけ）。

**運用上の注意**

- **賭けの締切**: 各取組に推定開始時刻(15:40〜17:52を等間隔配分)を割り当て、その5分前に個別ロック。結びの一番は17:47頃まで。締切はクライアント側の時刻で判定するので、データ登録のタイミングとは独立しています
- **翌日の前売り**は当日18:00から。翌日分の取組を `data/basho.json` に登録しておくと予想できます（未登録の間は「発表待ち」表示）
- 力士の一門は `index.html` 内の `ICHIMON_MAP` で定義。間違い・新入幕力士は自由に修正してください(未登録は自動割当)
- 力士名は番付と取組で表記を揃えてください（登録ツールが不一致を「要確認」と警告します）

## 追加SQL（番付編成会議対応・2026-07-13追加）

千秋楽終了時のスコアを凍結保存する列を追加します。SQL Editorで実行:

```sql
alter table public.players add column if not exists final_points integer;
alter table public.players add column if not exists history jsonb;
alter table public.players add column if not exists comment text;
```

`comment` はプレイヤーの「ひとこと」(番付表の四股名タップで吹き出し表示)です。

`history` は各プレイヤーの日次スコア推移(番付発表時の新聞記事風サマリー生成に使用)です。

これにより「千秋楽でスコア凍結 → 月・火は謹慎(ちゃんこ・クイズ休止) → 水曜10:00の番付編成会議で全員一斉に番付確定・発表」というフローになります。

## 追加SQL（みんなの賭け動向・2026-07-18追加）

「場所予想」タブに、取組予想でどの力士が人気か（全員分の賭け件数の集計）を表示するための列を追加します。SQL Editorで実行:

```sql
alter table public.players add column if not exists bet_trend jsonb;
```

`bet_trend` は各プレイヤーの「今場所、力士名ごとに何回賭けたか」の集計値です（例: `{"大の里": 5, "豊昇龍": 3}`）。ベットのたびに自動同期され、全員分を合算した人気ランキングが場所予想タブに表示されます。

この列を追加しなくてもゲームは動作します（賭け動向カードに「まだ賭けデータがありません」と表示されるだけです）。

## 追加設定（ログインID・パスワード対応・2026-07-15追加）

Googleアカウント等を使わず、ログインID・パスワードだけでアカウントを管理できるようにする設定です。実在するメールアドレスは一切収集しません(内部的に`ログインID@toto-sumo.local`という実在しない偽メールを組み立てて、Supabase Authの認証機構だけを流用しています)。

### SQL: 四股名の重複禁止

先に重複がないか確認してから制約を付けます。SQL Editorで実行:

```sql
-- 1. 重複がないか確認（0行なら次へ進んでOK）
select shikona, count(*) from public.players group by shikona having count(*) > 1;

-- 2. 重複がなければ一意制約を追加
alter table public.players add constraint players_shikona_unique unique (shikona);
```

もし1.で重複行が出た場合は、どちらかの四股名を手動で変えてから2.を実行してください。

### SQL: フルセーブ用の非公開テーブル

推し力士・実績・的中履歴・現在の未精算ベットなど、`players`テーブルには入れていない詳細データを別端末にも引き継ぐためのテーブルです。`players`は「誰でも閲覧可」のランキング表示用テーブルなので、個人の詳細データはそこに混ぜず、**本人しか読めない**別テーブルに分けています。SQL Editorで実行:

```sql
create table if not exists public.player_saves (
  id uuid primary key references auth.users(id) on delete cascade,
  full_state jsonb,
  updated_at timestamptz not null default now()
);

alter table public.player_saves enable row level security;

create policy "本人のみ閲覧可" on public.player_saves
  for select using (auth.uid() = id);

create policy "本人のみ作成可" on public.player_saves
  for insert with check (auth.uid() = id);

create policy "本人のみ更新可" on public.player_saves
  for update using (auth.uid() = id);
```

### Supabaseダッシュボードの設定変更

1. 「Authentication」→「Sign In / Providers」→「Email」を開き、**「Confirm email」をOFF**にする(偽メールは実在しないため、確認メールを送る方式では詰まってしまいます)
2. 同じ画面(または「Authentication」→「Providers」の匿名ログイン設定近く)で**「Allow manual linking」をON**にする(匿名アカウントにログインID・パスワードを後付けするために必要)

この2つを設定しないと、ゲーム内の「アカウント設定」からのログインID登録がエラーになります。

### 動作について

- 今まで通り匿名で始められ、入門の儀(または米取表タブの「アカウント設定」)でログインID・パスワードを設定すると、**同じ持ち米・称号のまま**永続アカウント化されます(既存プレイヤーのデータはそのまま引き継がれます)
- 設定後は、別の端末・別のブラウザから同じログインIDでログインし直せます。推し力士・的中履歴・実績・現在の未精算ベットを含め、**これまでのデータがすべて復元されます**(`player_saves`テーブルへの保存は、ベットや設定変更のたびに自動で行われます)
- ただし、ログインID設定を行った端末が`player_saves`テーブル追加**より前**に一度もアプリを開き直していない場合、その回だけは持ち米・称号などの集計値のみの簡易復元になります(元の端末を一度開けば、次回以降は全データが復元されます)
- 四股名は場所を問わず全プレイヤーで重複できません(先着順)

## 追加SQL（過去のトト番付アーカイブ・2026-07-15追加）

`players`テーブルは場所が変わるたびに上書きされるため、場所が終わるとその場所の最終順位はどこにも残りません。過去の場所を後から見返せるように、「場所キー×プレイヤー」で1行の記録用テーブルを追加します。番付編成会議(`announceBanzuke()`)のタイミングで、各プレイヤーが**自分の行だけ**を書き込みます(一斉スナップショット方式だと、開くのが遅い人の分が欠けたまま確定してしまうため、各自が自分のタイミングで書き込む方式にしています)。SQL Editorで実行:

```sql
create table if not exists public.basho_results (
  basho_key text not null,
  player_id uuid not null references auth.users(id) on delete cascade,
  shikona text,
  ichimon text,
  title text,
  final_points integer,
  hits integer,
  bets integer,
  banzuke_level integer,
  banzuke_pos integer,
  banzuke_total integer,
  updated_at timestamptz not null default now(),
  primary key (basho_key, player_id)
);

alter table public.basho_results enable row level security;

create policy "誰でも閲覧可" on public.basho_results
  for select using (true);

create policy "自分の行のみ作成可" on public.basho_results
  for insert with check (auth.uid() = player_id);

create policy "自分の行のみ更新可" on public.basho_results
  for update using (auth.uid() = player_id);

create index if not exists basho_results_key_points
  on public.basho_results (basho_key, final_points desc);
```

米取表タブの「過去のトト番付」アコーディオンを開くと、記録済みの場所がプルダウンで選べ、選んだ場所の最終順位表が表示されます。9月場所が始まって以降、水曜10時の番付編成会議を経た人から順に7月場所の記録が積み上がっていきます。

## 動作確認チェックリスト

- ゲームを開いてヘッダーに 🟢 が表示される(⚪ のままなら STEP 3・4 を再確認)
- Supabase の「Table Editor」→ `players` に自分の行が現れる
- 別のブラウザ(またはスマホ)で開いて別の四股名で入門 → 米取表タブのランキングに両方表示される
- 同じ四股名で入門しようとするとエラーになる
- 入門の儀でログインID・パスワードを設定 → 別のブラウザで「すでにログインIDをお持ちの方はこちら」からログイン → 持ち米・称号が復元される

## 運用メモ

- **進行**: 本場所は奇数月の第2日曜が初日(次回: 2026年7月12日〜26日)。場所間はオフシーズンで、ちゃんことクイズのみ
- **締切**: 毎日15:30(日本時間)に投票締切、16:00に結果発表。結果はアプリを開いたときに自動精算されます
- **データ**: ベット履歴・称号は各自のブラウザ(LocalStorage)、成績のみSupabase。ブラウザのデータを消すと手元の記録は消えます
- **不正について**: 成績は各クライアントが自己申告する信頼ベースです。友達同士で遊ぶ前提の設計で、本格運用するならEdge Functionでの検証追加を推奨
- **無料枠**: Supabase無料プランで数百人規模まで余裕です。1週間アクセスがないと一時停止されるので、ダッシュボードから再開してください

## トラブルシューティング

| 症状 | 対処 |
|------|------|
| ヘッダーが ⚪ のまま | CONFIG の URL / anonKey を再確認。ブラウザのコンソール(F12)にエラーが出ていないか確認 |
| 「匿名ログイン失敗」 | STEP 3 の Anonymous Sign-Ins が Enabled か確認 |
| ランキングに他の人が出ない | 相手がまだ今場所ベットしていない可能性。ベット後に反映されます |
| 昔のスマホで動かない | ES2020対応ブラウザが必要(Chrome/Safari/Edge の近年版) |
