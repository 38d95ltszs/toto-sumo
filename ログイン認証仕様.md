
# トト相撲 ONLINE ログイン認証仕様

## 概要

Googleアカウントなどの外部サービスを使わず、**ログインID・パスワードだけ**でアカウントを管理する。実在するメールアドレスは一切収集しない。認証基盤にはSupabase Authをそのまま使い、内部的に組み立てた偽メールアドレスを識別子として流用する方式（フェイクメール方式）を取っている。

匿名ログイン（Supabaseの`signInAnonymously()`）は引き続き基本の入口として残り、ログインID・パスワードは「その匿名アカウントに後付けで紐付ける」形で永続化する。これにより、既存の匿名プレイヤーもデータを失わずに移行できる。

## 認証方式の全体像

1. 初回起動時、プレイヤーは自動的に匿名ユーザーとしてSupabaseにサインインする（`sbInit()`）
2. 入門の儀、または米取表タブの「アカウント設定」で、任意でログインID・パスワードを設定できる
3. 設定すると、**同じUUID・同じ持ち米・同じ実績のまま**、匿名アカウントが永続アカウントに切り替わる（`linkCredentials()`）
4. 別の端末・別のブラウザからは、設定したログインID・パスワードで同じアカウントにログインし直せる（`signInWithLoginId()`）

## 偽メールアドレス方式

Supabase Authのパスワード認証は、識別子としてメールアドレスまたは電話番号を要求する。ユーザーには一切見せない内部専用の文字列として、次の形式を組み立てて使う。

```js
function fakeEmailOf(loginId) { return `${loginId.toLowerCase()}@toto-sumo.local`; }
```

`toto-sumo.local`は実在しないドメインであり、確認メールが届くことも、外部に送信されることもない。ユーザーが実際に入力するのは「ログインID」と「パスワード」のみ。

ログインIDのバリデーション:

```js
function validLoginId(s) { return /^[a-zA-Z0-9_]{4,20}$/.test(s); }
```

半角英数字とアンダースコアのみ、4〜20文字。パスワードは6文字以上を要求（`password.length < 6`でチェック）。

Supabase Auth内部では`auth.users.email`にこの偽メールが入る形になるが、この値はプレイヤー本人以外には見えない（後述のRLSにより`players`テーブルなど公開テーブルには一切含めていない）。

## 主要な関数

| 関数 | 役割 |
|---|---|
| `sbInit()` | Supabaseクライアントを初期化し、セッションが無ければ匿名サインインする。`MYID`（ユーザーUUID）と`IS_ANON`（匿名のままか）を設定する |
| `linkCredentials(loginId, password)` | 現在の（匿名）セッションに偽メール＋パスワードを紐付ける。`supabase.auth.updateUser({email})`→`updateUser({password})`の順に呼ぶ。同じUUIDのまま永続アカウント化されるため、既存の`players`行・`player_saves`行はそのまま引き継がれる |
| `signInWithLoginId(loginId, password)` | 別端末からの復帰ログイン。`supabase.auth.signInWithPassword()`でそのUUIDのセッションを取得し、`players`行と`player_saves`行（フルセーブ）を取得して返す |
| `hydrateG(obj)` | 復元したデータ（localStorageまたはクラウドのフルセーブ）から、ゲーム内部状態`G`を正規化して構築する共通処理 |

状態変数`IS_ANON`が`true`の間はログインID未設定（＝匿名のまま）を意味し、UIの「アカウント設定」カードの表示切り替えなどに使う。

## データの保存先（3テーブルの役割分担）

Supabase側は用途によって3つのテーブルに分けている。1つにまとめず分けている理由は、**公開して良い情報（ランキング表示用）と、本人以外に見せてはいけない情報（プレイ内容の詳細）を混在させないため**。

### `players`（公開・ランキング表示用）

RLS: 誰でも閲覧可、自分の行のみ作成・更新可。

四股名・一門・称号・場所キー・持ち米・的中数・ベット数・番付レベル・参加場所数・千秋楽時点の凍結スコアなど、米取表のランキングに必要な情報のみ。ログインIDやパスワードに関する情報は一切含まない。四股名には一意制約（`players_shikona_unique`）があり、全プレイヤーで重複できない。

### `player_saves`（非公開・本人のみ）

RLS: `auth.uid() = id`の行のみ閲覧・作成・更新可（他人からは一切読めない）。

ゲーム内部状態`G`をまるごと`full_state`（jsonb）として保存する。推し力士・実績一覧・的中履歴・現在の未精算ベット・優勝予想/三賞予想など、`players`テーブルに入れていない詳細データはすべてここに入る。ベットや設定変更のたびに自動保存され（`pushProfile()`内でupsert）、別端末からのログイン時にはこの内容がそのまま`G`に復元される（`hydrateG()`）。

### `basho_results`（公開・場所ごとのアーカイブ）

RLS: 誰でも閲覧可、自分の行のみ作成・更新可（`basho_key, player_id`の複合主キー）。

場所が締まって番付編成会議を迎えたタイミングで、各プレイヤーが**自分の行だけ**を書き込む（`announceBanzuke()`内）。過去の場所の最終順位を、場所が変わっても参照できるようにするための記録用。

## 画面・UIフロー

### 新規プレイヤー（入門の儀）

`renderOnboarding()`で四股名・一門・推し力士・ひとこと・（Supabase設定時のみ）ログインID・パスワードを入力する。送信（`onbStart()`）の流れ:

1. 四股名・入力内容のバリデーション
2. `sbInit()`で匿名セッションを確立
3. `players`テーブルに対し四股名の重複を事前チェック（重複していれば即エラー、送信不可）
4. ログインID・パスワードが入力されていれば`linkCredentials()`で紐付け（IDが既に使われていれば「そのログインIDは既に使われています」を表示）
5. `newGame()`でローカルのゲーム状態を作成し、以後は通常の起動フローへ

通信状況によりオンライン処理が失敗した場合でも、四股名・一門などのローカルプレイ自体はブロックされない（後で「アカウント設定」から再設定できる旨をトーストで案内）。

### 別端末からの復帰ログイン

入門の儀画面の「すでにログインIDをお持ちの方はこちら」から`renderSignIn()`に切り替わり、ログインID・パスワードのみを入力する（`onbSignIn()`）。

- `player_saves`にフルセーブがあれば、推し力士・実績・現在の未精算ベットを含め**全データを復元**する
- フルセーブがまだ無い場合（この仕組みを導入する前にログインID設定した端末を、導入後まだ一度も開いていない場合など）は、`players`テーブルの集計値（持ち米・称号など）のみの簡易復元になる。この場合はその旨をダイアログで案内する

### 既存の匿名プレイヤーへの移行導線

米取表タブに常設の「🔑 アカウント設定」カードがあり、`IS_ANON`が`true`の間はログインID・パスワードの設定フォームを表示する（`onSetupAccount()`）。設定済みの場合はログインIDを表示するのみ。

匿名のまま・未設定のプレイヤーには、起動時に一度だけ設定を促すトーストを表示する（`G.acctPromptShown`フラグで一度きりに制御）。

## Supabase側で必要な設定

コードの変更だけでは完結せず、Supabaseダッシュボード側で以下2つの設定が必須。

1. 「Authentication」→「Providers」→「Email」で **Confirm email をOFF**（偽メールは実在しないため、確認メール方式のままだと詰まる）
2. 「Authentication」→「Providers」で **Allow manual linking をON**（匿名アカウントへログインID・パスワードを後付けするために必要）

この2つが未設定だと、「アカウント設定」からのログインID登録がエラーになる。

## プライバシー・セキュリティ上の整理

- 実在するメールアドレス・電話番号・氏名は一切収集しない。ログインIDは利用者が自由に決める文字列であり、個人情報保護法上の「個人情報」には該当しない設計を意図している
- パスワードはSupabase Auth側でハッシュ化（不可逆変換）されて保存される。運営者を含め、誰も元のパスワードを見ることはできない
- 上記の内容は、入門の儀・アカウント設定画面の入力欄下に注記として表示している

## 既知の制限事項

- ログインID設定前（匿名のまま）にブラウザのデータを消すと、そのアカウントには二度とアクセスできなくなる（匿名ユーザーの一般的な制約）
- `player_saves`の同期は`save()`が呼ばれるたびに行われる非同期処理のため、ネットワーク切断中に閉じるなど極端なタイミングでは最新の1回分が反映されない可能性がある
- 別端末ログイン時、`player_saves`にフルセーブが無い（この仕組み導入前の端末をまだ開いていない）場合は集計値のみの簡易復元になる
